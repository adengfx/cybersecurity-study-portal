NIST Cybersecurity Framework (CSF) – Overview

The Six Core Functions

Govern
Establishes and monitors the org's cyber risk management strategy, policies and oversight.
Identify
Develops understanding of org assets, people, data, systems and risks.
Protect
Implements safeguards to ensure delivery of critical infrastructure.
Detect
Develops activities to identify the occurrence of a cybersec event.
Respond
Takes actions regarding a detected event, including mitigation and forensics.
Recover
Maintains plans for resilience and restores capabilities impaired by an incident.
CH 1

Cyber Threats

The Cyber Kill Chain (7 Stages)

01
Reconnaissance
Finding the target and understanding its characteristics.
02
Weaponisation
Customising a known vulnerability for a specific target; integrating into an automated platform. Criminals often purchase rather than build malware.
03
Delivery
Methods: phishing, compromised website, stolen credentials, exposed vulnerabilities, infected USB drive.
04
Exploitation
Via attachment, browser vulnerability, service protocol vulnerability, or unauthorised credentials.
05
Installation
Payload installed in memory/HDD. Registry entry used to persist across reboots.
06
Command & Control
Compromised system connects to C2 server, registers as a host and takes instructions (list/extract files, modify software etc.).
07
Action
Depends on motive: hacktivist = website defacement; criminal = financial gain; state-sponsored = exfiltrating specific files.

Cloaking & Alternate Data Streams

  • Malware can hide using Windows features, e.g. the hidden History folder at \AppData\Local\Microsoft\Windows\History.
  • desktop.ini contains cloaking entries: CLSID (hides from file-based listings) and UICLSID (hides in Windows Explorer).
  • NTFS files have a main $data stream; files can also have Alternate Data Streams (ADS). Windows only shows $data by default — malware hides in ADS.
  • Malware can also impersonate legitimate Windows processes (e.g. a keylogger using a keyboard hook callback routine hidden as a Windows process).

Rootkits

  • Installed via a dropper (e.g. malicious download). Not a virus; does not itself exploit vulnerabilities.
  • Can hide in the OS kernel, coded as a driver at ring 0 (deepest kernel level) using the Windows Driver Developer Kit.
  • Run with elevated privileges → can use Direct Kernel Object Modification (DKOM).
  • Often uses SSH for remote command & control — also encrypts downloaded malware to evade detection.
  • Real-world example: TDL3 (Dogma Millions group) — acts as a printer driver, installs an encrypted filesystem, per-install cost model. Downloads hidden payloads (keyloggers, DDoS tools etc.).

Remote Access Trojan (RAT)

  • A RAT (Remote Access Trojan) is malware that gives an attacker covert remote control over a compromised system.
  • A botmaster may deploy a RAT into a target to control it as part of a botnet.
  • Can be used to steal files, log keystrokes, activate webcams, or launch further attacks from the infected host.
  • Related to rootkits and C2 infrastructure — the RAT is the tool, the C2 server is the controller.

Stuxnet – State-Sponsored Cyber Attack

  • Stuxnet was the first widely known state-sponsored cyber weapon, discovered around 2010.
  • Jointly developed and launched by USA and Israel.
  • Targeted Iranian nuclear centrifuges (Siemens industrial control systems / SCADA), causing physical damage by making centrifuges spin at incorrect speeds.
  • A key real-world example of an APT with sabotage as its objective.

Phishing Variants & Watering Holes

  • Spear phishing — targeted at a specific person; researched and context-specific.
  • Mobile phishing — via SMS/mobile.
  • Vishing — phishing over VoIP.
  • Watering hole attack — a website trusted by a target group is compromised; malware downloads when visited. Attackers hope the target uses a business computer.

Advanced Persistent Threats (APT)

  • Malware directed at political/military targets using multiple attack vectors.
  • 5 Key Characteristics: Customised code · Focused objectives · Multi-threaded capabilities · Human-operated (not fully automated) · Low-and-slow operation to stay stealthy.
  • APT Objectives: Espionage · Sabotage.
CH 2

Frameworks for Protection Against Cyber Threats

The Orange Book (Early Security Concepts)

  • Published by the US Department of Defense as part of the Rainbow Series.
  • Describes a computer trusted to handle both classified and unclassified information.
  • Introduces 4 concepts: Reference monitor (mediates access to system resources) · Formal security model (for reading/writing info) · Trusted Computing Base (TCB) (subsystem containing all security code) · Assurance levels.

Orange Book – Assurance Levels (Divisions)

The Orange Book defines four divisions (D, C, B, A) with increasing security requirements.

LevelNameKey Requirement
DMinimal ProtectionNo security requirements met. Lowest level.
C1Discretionary ProtectionBasic user/data separation. Discretionary access control.
C2Controlled AccessStricter login, audit trails, resource isolation.
B1Labelled SecurityMandatory access control (MAC) for some objects. Security labels applied.
B2Structured ProtectionMandatory access control across the entire system. Formal security model enforced.
B3Security DomainsReference monitor fully implemented. Highly resistant to penetration.
A1Verified DesignFormal mathematical verification of design. Highest assurance.

NIST Cybersecurity Framework (CSF) – Detail

  • De facto standard for cybersecurity. Three elements: Framework Core · Framework Profile · Implementation Tiers.
  • Core decomposes into 22 security activities across the 6 functions.
  • Framework Profile aligns business outcomes with cybersec activities. Two profiles: Current Risk Profile (what cybersec looks like now) and Target Risk Profile (acceptable level of risk). A security plan bridges the gap.

Implementation Tiers (Maturity Levels)
TierNameKey Characteristics
1PartialAd hoc/reactive. No risk objectives. Little external collaboration.
2Risk InformedRisk practices formalised but not enterprise-wide. Internal sharing only.
3RepeatableRisk management mandated as policy. Internal and external collaboration.
4AdaptiveContinuous improvement based on changes to assets, threats and vulnerabilities.

Key Implementation Steps (new cybersec program)
  • Identify business outcomes → understand threats/vulnerabilities → create current profile → conduct risk assessment → create target profile → determine/prioritise gaps → implement plan.

Cyber Risk Management

  • Key standards: NIST SP 800-30 (risk assessments) and ISO 27005 (information security risk management).
  • Start with threat intelligence — MITRE ATT&CK is a key catalogue of threats.
  • Risk treatments: Acceptance · Avoidance · Transfer · Mitigation.
  • Risk Register — used to record and track risks (usually a spreadsheet or automated tool).

IRAM2 – ISF Information Risk Assessment Methodology

IRAM2 is the risk methodology underlying the ISF Standard of Good Practice (SoGP). It has 6 phases:

01
Scoping
Define which systems and business processes are in scope.
02
Business Impact Assessment
Understand the effect on the business if a risk is realised.
03
Threat Profiling
Identify and profile relevant threats to the in-scope environment.
04
Vulnerability Assessment
Identify weaknesses that could be exploited by those threats.
05
Risk Evaluation
Combine threat and vulnerability findings to determine risk levels.
06
Risk Treatment
Decide how to treat each risk: accept, avoid, transfer or mitigate.

  • Categorise: System characteristics, CIA impact analysis, business endorsement.

COBIT Framework

  • Ensures enterprises have an effective, auditable set of IT processes that deliver value to stakeholders. Used by the financial sector for IT general controls audits.
  • Designed around process, grouped under: Plan (14) · Build (11) · Deliver (6) · Monitor (4).
  • Two key security processes: APO13 (managed security — Plan) and DSS05 (managed security — Deliver).
  • APO13 goals: IT/business compliance · IT/enterprise risk management · transparency of IT costs/benefits · security of info/infra/apps · reliable info for decision making.
  • APO13 control objectives: .01 Establish/maintain ISMS · .02 Define/manage a security plan · .03 Monitor and review ISMS.
  • Links to ISO 27001 (ISMS requirements).
  • DSS05 – 7 Objectives: .01 Protect against malware · .02 Network/connectivity security · .03 Endpoint security · .04 User identity/logical access · .05 Physical access to IT · .06 Sensitive documents/output devices · .07 Monitor infrastructure for security events.

Cybersecurity Controls

  • Controls must be cost effective. Implementing controls leaves residual risk.
  • 4 types of control (use 2 or more): Deterrent (reduces threats) · Preventative (reduces vulnerabilities) · Detective (identifies incidents) · Corrective (reduces impacts).
  • Two major control frameworks: NIST SP 800-53 and ISO 27002. Both use control categories; can be mapped against each other.
  • ISO 27002 hierarchy: Security Category → Control Objective → Control.
  • SP 800-53 hierarchy: Security Category → Control.
  • Statement of Applicability (SoA) — main link between risk and control set. Lists all controls considered; omitted controls documented with rationale.
  • ISF Standard of Good Practice — consistent with ISO 27002, NIST CSF, PCI DSS and GDPR. Risk framework: Business Impact → Threat/Vulnerability Assessment → Control Selection.
CH 3

Making Cybersecurity Effective

Security Architecture – SABSA

  • Rather than adopting a generic control set, orgs can use enterprise security architecture to create a solution that meets specific business needs.
  • SABSA captures business requirements and determines the security needed to meet them.
  • SABSA Matrix layers: Contextual · Conceptual · Logical · Physical · Component. (Top 2 = strategy/planning; bottom 3 = designing secure IT systems.)
  • CIA Triad (Confidentiality · Integrity · Availability) — important but not exhaustive. Can add non-repudiation, utility, authenticity.
  • SABSA attributes grouped into 7 categories: User · Management · Operational · Risk · Legal · Technical · Business.

Payment Card Industry Data Security Standard (PCI DSS)

  • Issued by the PCI Council to protect cardholder data. Reinforced by GDPR liability.
  • Key terminology: Merchant · Service Provider · Qualified Security Assessor (QSA) · Internal Security Assessor · Cardholder data (account number, cardholder name, expiry, service code) · Sensitive auth data (chip/swipe data, CVV, PIN).
  • v3.2 provides 12 requirement areas covering ~200 mandatory controls.
  • Key controls: Effective firewall config (rules reviewed every 6 months) · Network diagram maintained · Non-cardholder traffic denied · DMZ required for internet-facing systems · Default passwords changed · Stored cardholder data minimised (account numbers masked — show last 4 only) · Account numbers encrypted at rest · Transmitted data encrypted (SSL banned after Heartbleed) · AV software required on all systems · Security patches within 1 month · Source code review for custom dev (check for SQLi, XSS) · Production account numbers never used in testing.
  • Heartbleed — vulnerability in OpenSSL; allowed reading of memory from systems using vulnerable SSL/TLS. Reason SSL was deprecated by PCI.
  • In event of breach: investigation launched → safe harbor or fine issued → remediation applied → customers may be compensated.

Cloud Security

  • Three deployment models: IaaS · PaaS · SaaS.
  • Standards: ISO 27017 (cloud security controls) · NIST SP 800-144 (security/privacy in public cloud).
  • Main reference: Cloud Security Alliance (CSA) Security Guidance — 14 areas of control.
  • CSA Cloud Controls Matrix (CCM) v3.0.1 — 133 controls across 16 domains. Clarifies shared responsibilities between cloud provider and customer. Cross-references other major security standards.

Internet of Things (IoT) Security

  • IoT = everything connected to the internet that isn't a standard laptop, workstation or server. Can range from low-power 10-year sensors to industrial devices.
  • 2017 IoT Security Guidelines cover: Legal · Privacy · Security · Resilience · Survivability · Development.
  • No single definitive set of controls; guidance from OWASP, GSMA, and the IoT Security Foundation (142 controls, 13 areas).

Ensuring Security is Effective

  • Most breaches occur on systems with security controls — due to poor design, poor org hygiene, or new exploits.
  • In-house assurance: Vulnerability scans · Security controls audits · Penetration testing.
  • Independent assurance: Third-party evaluation, typically for certification (e.g. ISO 27000, Common Criteria).

Spectre & Meltdown

  • Hardware-level CPU vulnerabilities disclosed in January 2018, affecting almost all modern processors.
  • Meltdown — allows a process to read kernel memory it shouldn't have access to.
  • Spectre — tricks other processes into leaking their memory contents via speculative execution.
  • Despite widespread media coverage, no confirmed real-world attacks on businesses were recorded — largely a theoretical/proof-of-concept threat at the time.
  • Patches were released by OS vendors and CPU manufacturers, but with performance implications.

GDPR & Privacy

  • Enforces protection of privacy of EU citizens. Applies to any service consumed in the EU regardless of origin. Fine: up to 4% global revenue (notes say 1% — the higher figure is the standard maximum for major violations).
  • 6 Principles: Lawful/fair/transparent processing · Purpose-specific collection · Collect only what's needed · Keep data accurate · Store safely · Protect data.
  • GDPR requirements: Appropriate technical/org measures · Effective cybersec · Compliance (risk assessments, continuous monitoring).
  • Other practices: Data minimisation · Data Protection Impact Assessments (DPIA) · Privacy by design.
  • Key cybersec controls: Encryption · Access control · Audits · Data masking.

Zero Trust

  • New ABC: Assume nothing · Believe no one · Check everything.
  • Security mechanisms should assume any access request could be malicious.
  • NIST: Moves defences from static network-based perimeters to users, assets and resources.
  • Microsoft: Every access request is fully authenticated, authorised and encrypted before granting access.
  • CrowdStrike: All users continuously authenticated, authorised and security-config validated.
  • Concept first emerged in 2011. Internal networks cannot be trusted; threats are present at all times.
  • Mechanisms: Just-In-Time (JIT) access · Just Enough Access (JEA) · Tokenisation/encryption · Adaptive policies.

DORA – Digital Operational Resilience Act

  • Five security requirements: Cyber risk management · Cyber incident management · Resilience testing · Intelligence sharing · Third-party risk management.
  • Cyber Risk Management: Risk management body responsible for strategy, policies, procedures. Must approve business continuity plans, incident response plans, third-party services and audits. Annual review of risk framework. Continuous risk assessment.
  • Business Continuity: Plans must include business impact assessments and be regularly tested. Central security depositories require geographically separated secondary processing site.
  • Incident Management: Detect, classify (duration, asset criticality, economic impact), report. Major incidents must be reported to authorities.
  • Resilience Testing: Vulnerability assessments, scans, open-source analysis, security assessments, source code reviews, scenario/compatibility testing.
  • TLPT (Threat-Led Penetration Testing) — introduced by DORA. Accredited testers; carried out at least every 3 years.

Supply Chain Security & SBOMs

  • SBOM (Software Bill of Materials) — lists software components to enable supply chain tracking.
  • Required fields: Vendor name · Component name/version · Dependencies · Timestamps.
  • Best practices: New SBOM with each release · Automation essential at scale · Common formats: SPDX and CycloneDX.
  • Minimum requirements: Vulnerability database entries · Component licences · SaaS SBOM attestation.
  • 2021 US executive order mandated vendors provide SBOMs.
CH 4

Detecting and Managing Cyber Incidents

Incident Management Fundamentals

  • NICE Framework defines incident response roles: Cyber Defence Analyst · Incident Responder · Forensic Analyst.
  • US-CERT incident categories: Unauthorised access · Denial of service · Malware infections · Suspicious activities.
  • FIRST (Forum of Incident Response and Security Teams) — coordinates global incident response and information sharing. Established by CERT/CC (Carnegie Mellon University) along with a group of early CERTs.
  • National CERTs also play vital roles in coordinating incident response locally.
  • Incident handling phases: Preparation · Detection · Response · Follow-up (communication essential throughout).
  • NIST CSF response categories: Planning · Communication · Analysis · Mitigation · Improvements + Recovery.

Incident Response Maturity

  • CREST UK maturity assessment tool used to gauge readiness.
  • Preparation: form and train an IR team · establish processes · acquire tools.
  • Define procedures and playbooks — tools like TheHive support collaborative investigations.
  • Leverage strategic, tactical and operational threat intelligence.
  • Build relationships within the IR community and internally with IT teams.
  • Cyber crisis exercises test and improve response plans.

Detecting an Attack

  • Incidents should be prioritised by impact — not first-come-first-served.
  • SOC analysts monitor real-time alerts, scan for malware, watch for suspicious movements and manage false positives.
  • Incident response phases: Detection & analysis → Containment → Eradication → Recovery → Post-incident activities.

Threat Hunting

  • Proactive process: analysts search for hidden threats via suspicious activity, unusual files, unauthorised accounts.
  • Indicators of Compromise (IoCs) — essential clues that identify known threats and suspicious behaviour.
  • Advanced analytics on large datasets reveal hidden activity like beaconing to C2 servers.
  • Process: continuous monitoring → investigation → action to mitigate.
  • Tool: Malcom — real-time monitoring and analysis of captured network data.

Responding to an Incident

  • Early containment is critical — pre-authorised, tailored response plans make decisions faster.
  • Parallel activities: containment · recovery · ongoing investigation.
  • Eradication must address all affected systems, not just the initial compromise.
  • Continuous monitoring after incident — the initial attack may be a diversion.
  • Document and preserve evidence, especially if legal action may follow.

Communications Plan & Notification

  • Executive readiness: CEOs and board chairs must be prepared to face media and regulators. Clear roles and regular system checks.
  • Transparency: Clearly communicate what happened, its impact, and how affected parties can mitigate risks. Accept responsibility; don't downplay or blame others.
  • Preparation is crucial: Develop and rehearse a comms plan before incidents occur. Long-term goals include protecting brand and stock value.
CH 5

Cybersecurity and AI

AI Security Fundamentals

  • Training feeds large datasets into a model to refine probabilities; separate testing and validation datasets ensure accuracy.
  • Input → words converted to numerical tokens → pass through complex layers (vectors) → response generated.
  • Real-world AI risks: toxic outputs · hallucinated information · prompt injection · code injection.
  • Security must be applied throughout the AI lifecycle: protect training data · the model itself · validation data.

AI Threats

  • Supply chain attacks — compromised components or datasets undermine AI models.
  • Attacks on the model itself: inserting malicious code · extracting sensitive training data.
  • Traditional threats apply: ransomware · data poisoning.
  • AI-specific risks: toxic/hallucinated outputs · prompt injection (manipulating model responses).

AI Application Guardrails

  • Programmable, rule-based filters that sit between users and AI models to enforce application owner policies.
  • Validate user prompts before passing to AI; check AI responses before delivering to users.
  • Refuse to engage with harmful or demeaning inputs; provide neutral responses instead.

ISO 42001 – AI Management System

  • High-level framework for managing AI systems: secure, safe and responsible AI use.
  • Covers: principles · exceptions · resources · impact assessment · development.
  • AI impact assessments evaluate business and societal impacts, feeding into AI risk assessments.
  • Emphasises roles and responsibilities for both safety and security; requires detailed documentation of data, tools, systems and personnel.
  • Guidance on data provenance, acquisition, quality, preparation and security threats linked to data-dependent AI systems.
  • Supports documentation throughout the AI lifecycle; promotes transparency for stakeholders.
CH 6

Cyber Diplomacy

Cybersecurity Goes Global

  • UK initiated global conferences on cyberspace to address fears of the internet fracturing into separate national networks.
  • Russia and China have taken steps to control their national internet spaces (info control, cultural protection).
  • Internet technical standards managed by IETF; addressing managed by ICANN (US-based — source of international concern).
  • 7 proposed internet principles: Proportionality · Accessibility · Respect for diversity · Human rights · Openness · Collaboration against cybercrime · Fair competition.

UN Cyber Norms – The 11 Norms (UN GGE 2015)

  • There are 11 UN Cyber Norms, established by the UN Group of Governmental Experts (GGE) in 2015.
  • Define generally accepted rules for nation-states' internet behaviour; promote cooperation for global cybersecurity and stability.
  • Key norms: States' duty of care in incident handling · No malicious cyber activity from their territory · Protect critical infrastructure · Respect human rights (privacy, freedom of expression) · Share vulnerability information · Cooperate against cybercrime/terrorism · Don't attack other states' CERTs/CSIRTs · Transparency and confidence-building · Capacity building support for developing states.

GFCE & Cybil

  • GFCE (Global Forum on Cyber Expertise) — strengthens global cybersec capacity building by coordinating governments, private companies, civil society, tech industry and academia.
  • Cybil — portal used to share expertise and support countries in developing cybersec capabilities.
  • 5 GFCE themes: Cybersec policy & strategy · Incident management & critical infrastructure protection · Countering cybercrime · Cybersecurity culture & skills · Cybersecurity standards.

Traffic Light Protocol (TLP)

A scheme to manage how cybersec information is shared globally, balancing free flow with security.

ColourSharing LevelExample
● WHITE Freely shareable — no restrictions. General awareness material.
● GREEN Share widely within recipient's sector but not publicly. Sector-specific malware analysis.
● AMBER Share within recipient's org and trusted clients only. Sensitive indicators of compromise.
● RED Strictly for the named recipient only. Unauthorised sharing could cause harm. Highly sensitive intelligence.
